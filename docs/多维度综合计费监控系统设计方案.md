# 智衡(Balancer) - 多维度综合计费监控系统设计方案

## 一、设计概述与核心目标

### 1.1 项目背景
智衡(Balancer)需要支持监控多种复杂的计费模式，特别针对类似"方舟coding plan"这类**多时间维度、嵌套配额**的综合计费方案。系统需同时支持传统金钱余额监控与复杂配额管理。

### 1.2 核心设计挑战
1. **多层嵌套配额**：五小时、周、月三个时间维度的配额相互关联，使用量需同时扣除
2. **独立刷新机制**：每个时间维度有独立的刷新周期和重置逻辑
3. **综合计费策略**：多种计费模式（金钱、Tokens、订阅）的统一管理
4. **实时监控与预警**：多层级的余额状态监控和预警机制

### 1.3 设计原则
1. **统一抽象模型**：将不同计费模式抽象为统一的额度管理模型
2. **灵活扩展性**：支持未来新增计费维度和策略类型
3. **数据一致性**：确保多层额度扣减的原子性和一致性
4. **用户体验优先**：直观展示复杂配额关系和使用状态

## 二、核心概念定义

### 2.1 额度维度 (Quota Dimension)
系统的基本管理单元，代表一个特定时间周期的使用配额：

| 维度类型 | 时间周期 | 刷新规则 | 嵌套关系 |
|---------|---------|---------|---------|
| 小时级配额 | 5小时 | 每5小时重置 | 计入周、月维度 |
| 周级配额 | 7天 | 每周固定时间重置 | 计入月维度 |
| 月级配额 | 30天 | 每月固定日期重置 | 顶级维度 |
| 永久配额 | 永久 | 永不刷新 | 独立维度 |
| 金钱余额 | 实时 | 按需充值 | 独立维度 |

### 2.2 额度策略组 (Quota Strategy Group)
管理相关额度维度的集合，特别针对综合计费方案：
- **综合计费组**：包含小时、周、月三个嵌套维度
- **独立计费组**：单一维度策略（如纯金钱余额）
- **混合计费组**：多种计费模式的组合

### 2.3 使用扣减规则 (Usage Deduction Rules)
定义额度消耗时的处理逻辑：
1. **全维度检查**：使用前检查所有相关维度的可用余额
2. **原子扣减**：同时从所有相关维度扣除使用量
3. **优先级处理**：当某一维度余额不足时的处理策略

### 3.2 详细数据表设计

#### 3.2.1 账户表 (accounts)
存储账户基础信息，支持多种认证方式和服务提供商集成。

#### 3.2.2 额度策略组表 (quota_strategy_groups)
```sql
CREATE TABLE quota_strategy_groups (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    account_id INTEGER NOT NULL,
    name VARCHAR(255) NOT NULL,                 -- 策略组名称
    strategy_type VARCHAR(50) NOT NULL,         -- 类型：comprehensive, independent, mixed
    description TEXT,                           -- 描述
  
    -- 状态管理
    is_active BOOLEAN DEFAULT TRUE,
    priority INTEGER DEFAULT 0,                 -- 使用优先级

    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
```

#### 3.2.3 额度维度表 (quota_dimensions)
```sql
CREATE TABLE quota_dimensions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    group_id INTEGER NOT NULL,
  
    -- 维度标识
    dimension_name VARCHAR(100) NOT NULL,       -- 维度名称：5h_quota, weekly_quota等
    dimension_type VARCHAR(50) NOT NULL,        -- 类型：time_based, permanent, monetary
    unit VARCHAR(50) NOT NULL,                  -- 单位：requests, tokens, CNY, USD等
  
    -- 配额配置
    total_quota DECIMAL(20, 6) NOT NULL,        -- 总配额
    current_balance DECIMAL(20, 6) NOT NULL,    -- 当前余额
    reserved_balance DECIMAL(20, 6) DEFAULT 0,  -- 预留余额（用于预扣费）
  
    -- 时间周期配置
    time_period VARCHAR(20),                    -- 时间周期：5h, 7d, 30d, permanent
    reset_time VARCHAR(10),                     -- 重置时间（如"08:00"）
    reset_day INTEGER,                          -- 重置日（1-31，周维度时为星期几）
    timezone VARCHAR(50) DEFAULT 'UTC',         -- 时区
  
    -- 刷新状态
    last_reset_at TIMESTAMP,                    -- 上次重置时间
    next_reset_at TIMESTAMP,                    -- 下次重置时间
    auto_reset BOOLEAN DEFAULT TRUE,            -- 是否自动重置
  
    -- 关联关系
    parent_dimension_id INTEGER,                -- 父维度ID（用于嵌套关系）
    nesting_level INTEGER DEFAULT 0,            -- 嵌套层级：0-顶级，1-子级，2-孙级
  
    -- 告警阈值
    warning_threshold DECIMAL(20, 6),           -- 警告阈值（绝对值）
    warning_percentage DECIMAL(5, 2),           -- 警告阈值（百分比）
    critical_threshold DECIMAL(20, 6),          -- 严重阈值
    critical_percentage DECIMAL(5, 2),          -- 严重阈值百分比
  
    -- 使用限制
    max_usage_per_reset DECIMAL(20, 6),         -- 每周期最大使用量
    min_balance_required DECIMAL(20, 6),        -- 最小保留余额
  
    FOREIGN KEY (group_id) REFERENCES quota_strategy_groups(id) ON DELETE CASCADE,
    FOREIGN KEY (parent_dimension_id) REFERENCES quota_dimensions(id),
  
    INDEX idx_dimension_group (group_id),
    INDEX idx_dimension_type (dimension_type),
    INDEX idx_reset_time (next_reset_at),
    UNIQUE (group_id, dimension_name)
);
```

## 四、业务逻辑设计

### 4.1 额度检查与扣减流程

#### 4.1.1 预检查流程
```
1. 接收使用请求
   ├── 账户验证
   ├── 服务端点识别
   └── 预估使用量计算

2. 维度余额检查
   ├── 识别相关额度维度
   ├── 检查各维度可用余额
   ├── 验证嵌套关系约束
   └── 预留额度（可选）

3. 可用性判断
   ├── 所有维度充足 → 允许使用
   ├── 部分维度不足 → 按策略处理
   └── 关键维度不足 → 拒绝使用
```

#### 4.1.2 原子扣减事务
```typescript
// 伪代码示例，不包含实际代码
class AtomicDeductionService {
  async deductUsage(accountId, usageRequest) {
    // 1. 开启事务
    const transaction = await beginTransaction();
  
    try {
      // 2. 锁定相关维度记录
      const dimensions = await lockDimensions(accountId, usageRequest);
    
      // 3. 验证所有维度余额充足
      for (const dim of dimensions) {
        if (!await hasSufficientBalance(dim, usageRequest.estimatedUsage)) {
          throw new InsufficientBalanceError(dim.dimension_name);
        }
      }
    
      // 4. 执行扣减（按嵌套关系调整）
      const deductionResults = [];
      for (const dim of dimensions) {
        const result = await performDeduction(dim, usageRequest, dimensions);
        deductionResults.push(result);
      }
    
      // 5. 记录使用详情
      const usageRecord = await createUsageRecord(accountId, usageRequest, deductionResults);
    
      // 6. 提交事务
      await commitTransaction(transaction);
    
      return {
        success: true,
        transactionId: transaction.id,
        remainingBalances: deductionResults.map(r => ({
          dimension: r.dimensionName,
          balance: r.newBalance
        }))
      };
    
    } catch (error) {
      // 7. 回滚事务
      await rollbackTransaction(transaction);
      throw error;
    }
  }
}
```

#### 4.1.3 嵌套维度扣减策略
1. **完全同步扣减**：使用量同时从所有相关维度扣除相同数值
2. **比例扣减**：根据预设比例在不同维度间分配扣减量
3. **层级优先扣减**：优先扣除子维度余额，不足时使用父维度
4. **智能分配扣减**：根据各维度余额比例动态分配

### 4.2 额度刷新机制

#### 4.2.1 多维度协同刷新
```
刷新调度器 (RefreshScheduler)
├── 时间维度刷新
│   ├── 5小时维度：每5小时触发
│   ├── 周维度：每周固定时间触发
│   └── 月维度：每月固定日期触发
├── 依赖关系处理
│   ├── 刷新顺序控制（父维度先于子维度）
│   ├── 刷新冲突解决
│   └── 刷新失败重试
└── 状态同步
    ├── 刷新记录保存
    ├── 余额重置通知
    └── 历史数据归档
```

#### 4.2.2 刷新策略配置
1. **硬性重置**：余额完全重置为总配额
2. **软性重置**：保留部分未使用余额（可累积）
3. **动态重置**：根据历史使用模式调整配额
4. **手动重置**：管理员或用户手动触发重置

### 4.3 余额计算与统计

#### 4.3.1 实时余额计算
1. **当前可用余额** = `当前余额 - 预留余额`
2. **使用率** = `(总配额 - 当前余额) / 总配额 × 100%`
3. **预估耗尽时间** = `当前余额 / 平均使用速率`

#### 4.3.2 跨维度统计
1. **综合使用率**：加权计算多维度整体使用率
2. **瓶颈维度识别**：找出最先耗尽的维度
3. **使用模式分析**：分析各维度使用的时间分布

## 六、告警与通知系统

### 6.1 多层级告警规则

#### 6.1.1 维度级告警
```yaml

  - 余额低于10次：立即通知
  - 余额低于20次：提前1小时通知
  - 使用率超过80%：预警通知

周维度告警：
  - 余额低于50次：立即通知
  - 余额低于100次：提前24小时通知
  - 日使用量异常：统计分析告警

月维度告警：
  - 余额低于200次：立即通知
  - 余额低于500次：提前7天通知
  - 周使用趋势异常：趋势分析告警
```

#### 6.1.2 综合状态告警
1. **瓶颈预警**：任一维度即将耗尽时通知
2. **使用失衡**：各维度使用率差异过大时提醒
3. **刷新提醒**：重要维度刷新前通知
4. **配置异常**：检测到不合理配置时告警

### 6.2 智能通知策略

#### 6.2.1 通知优先级
```
P0 - 紧急告警（服务即将中断）
  ├── 任一维度余额为0
  ├── 关键API调用失败
  └── 系统异常

P1 - 重要告警（需要立即关注）
  ├── 余额低于严重阈值
  ├── 使用量异常激增
  └── 刷新失败

P2 - 警告通知（需要关注）
  ├── 余额低于警告阈值
  ├── 使用率过高
  └── 配置变更

P3 - 信息通知（仅供参考）
  ├── 余额正常刷新
  ├── 定期报告
  └── 使用趋势分析
```

### 7.2 数据流设计
```
实时数据流：
1. API调用 → 使用记录收集 → 配额检查 → 扣减执行
2. 余额变更 → 状态更新 → 告警检查 → 通知触发

批量处理流：
1. 定时任务 → 数据聚合 → 报表生成 → 存储归档
2. 历史分析 → 模式识别 → 策略优化 → 配置调整
3. 系统维护 → 数据清理 → 备份执行 → 健康检查
```

### 7.3 缓存与性能优化
```
多级缓存策略：
├── L1缓存：内存缓存（余额状态、配置信息）
├── L2缓存：Redis集群（使用记录、统计结果）
└── L3缓存：本地数据库（历史数据、配置持久化）

性能优化措施：
1. 异步处理：非关键操作异步执行
2. 批量操作：合并小额度扣减请求
3. 读写分离：主从数据库架构
4. 数据分片：按账户ID分片存储
5. 预计算：常用统计指标预计算缓存
```

## 十、总结

### 10.1 方案优势
1. **全面性**：覆盖从简单金钱余额到复杂嵌套配额的所有计费模式
2. **灵活性**：可配置的时间维度、扣减策略和告警规则
3. **实时性**：毫秒级余额检查和原子扣减保证数据一致性
4. **智能性**：基于机器学习的预测和优化建议
5. **扩展性**：模块化设计和开放API支持快速扩展

### 10.2 关键技术亮点
1. **多维度嵌套配额模型**：业界首创的复杂配额关系管理方案
2. **原子扣减事务**：保证多维度余额扣减的一致性
3. **协同刷新调度**：智能处理多时间维度的刷新冲突
4. **智能预测引擎**：基于多因素的综合耗尽时间预测
5. **可视化关系展示**：直观展示复杂配额嵌套关系

### 10.3 商业价值
1. **成本控制**：精准监控和预警，避免意外超额费用
2. **资源优化**：智能配额分配，最大化资源利用率
3. **风险防范**：多层级告警系统，提前发现潜在问题
4. **决策支持**：深度数据分析，为采购和预算提供依据
5. **效率提升**：自动化监控和管理，减少人工操作成本

本设计方案为智衡(Balancer)提供了一个完整、先进的多维度综合计费监控解决方案，能够满足最复杂的计费场景需求，同时保持系统的灵活性、可扩展性和用户友好性。
